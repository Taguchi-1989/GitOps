# FlowOps 利用者ペルソナと UX 評価

ブランチ `claude/improve-ux-accessibility-dTezA` のUX/A11y改善が「実際の利用者が困らない」状態になっているかを4ペルソナで検証する。
各ペルソナごとに、ダッシュボード → 課題作成 → 改善案適用までの主動線を辿り、改善後の挙動と残課題を記す。

---

## ペルソナ1: 田中 美咲 (32) / 業務改善担当 (現場)

| 項目 | 内容 |
| --- | --- |
| 職種 | 営業事務リーダー (非IT職) |
| ITリテラシー | Excel/Slack/Notion は使える。Git/YAML は未経験 |
| 利用シーン | 月1回、業務フローの非効率な箇所を見つけて改善提案 |
| デバイス | 会社支給ノートPC (Windows/Chrome) + 移動中は iPhone |
| ゴール | 「承認フローのこのステップを飛ばしたい」を上司に伝えて反映してもらう |
| 困りごと | 専門用語(マージ/トリアージ/ブランチ)がわからず固まる、入力ミスで赤くなると焦る |

### 動線評価
1. **初回訪問** → WelcomeGuide が自動で開く。「ノード」「ブランチ」に括弧で言い換え (例: `フロー上のひとつのステップ`、`作業スペース(Gitブランチ)`) を入れたので**用語の壁が下がった**。 ✅
2. **ダッシュボード** → ステータスバッジ「トリアージ」にホバーで `内容を確認して優先度や担当を決める段階` が出る。マウス操作なら問題なし。 ✅
3. **課題を作成** → 「良い課題の書き方」アサイドが見える。タイトル空欄で送信すると赤枠+`タイトルを入力してください` がフィールド直下に出る。Toast「入力内容を確認してください」も読みやすい。**焦らず修正できる**。 ✅
4. **モバイル(iPhone)で見直し** → ハンバーガー化で本文が読める。タップ領域 44px 確保で誤タップ減。 ✅

### 残課題
- WelcomeGuide を閉じた後に再表示する方法 (サイドバー「使い方ガイド」) がモバイルで Drawer を開かないと見えない。発見しづらい。
- 「Issue」という英語表記がフロー画面の遷移メッセージにまだ残っている箇所がある可能性 (要追加調査)。

## ペルソナ2: 佐藤 健一 (48) / 部長

| 項目 | 内容 |
| --- | --- |
| 職種 | 業務統括部長 (元SE、現在は管理職) |
| ITリテラシー | Git は触ったことがあるが日常的には使わない |
| 利用シーン | 部下が起票した課題を週次でレビュー、承認/差戻し |
| デバイス | デスクトップ27インチ + iPad Pro (会議中) |
| ゴール | 現場の声を素早く把握し、適切に判断する |
| 困りごと | 老眼で小さい文字がつらい、AIの判断根拠が見えないと承認を渋る |

### 動線評価
1. **27インチで一覧表示** → ダッシュボードのStatカードが見やすい。フォーカスリングが明示されてキーボード操作も追える。 ✅
2. **コントラスト** → `text-gray-500/400` を `text-gray-700/300` に底上げしたので、老眼でも読める明度になった。 ✅
3. **iPad横向き** → サイドバーは md ブレークポイント(768px)以上で常時表示。iPad Pro 横向きなら問題なし。縦向きはハンバーガーに切替。 ✅
4. **ConfirmDialog の「この操作で何が起きるか」** → 承認前に何が変わるかを箇条書きで提示。判断材料が揃う。**心理的安全性が高い**。 ✅

### 残課題
- AI改善案の生成根拠 (なぜこの提案か) は DiffPreview に表示されているが、現状コンテキストでは確認していない。要レビュー対象。
- StatCardの値(数字)が `text-3xl` だが、本文 `text-sm` (14px) は老眼にはまだ小さめ。`text-base` (16px) 化を将来検討。

## ペルソナ3: 山田 翔 (24) / 入社2年目 SE

| 項目 | 内容 |
| --- | --- |
| 職種 | 社内SE (情シス) |
| ITリテラシー | 高い (Git/CLI/Docker いける) |
| 利用シーン | YAMLを直接編集してフロー定義を整える、API経由で連携 |
| デバイス | MacBook Pro + 外部ディスプレイ。VS Code と並べて使う |
| ゴール | 効率よく多数の課題を捌く、ショートカット重視 |
| 困りごと | モーダルがマウス操作前提だとストレス、無駄なクリックが多いと萎える |

### 動線評価
1. **キーボードのみで操作** → Tab で Skip link が最初に出る → メインへジャンプ可能。サイドバーをスキップできるので速い。 ✅
2. **WelcomeGuide を矢印キー(←→)で前後** → 実装済み。Escape で閉じる。 ✅
3. **ConfirmDialog で Escape キャンセル** → 破壊的操作を回避しやすい。初期フォーカスがキャンセル側なので Enter 暴発もなし。 ✅
4. **IssueList のタブ切替** → WAI-ARIA tab pattern (←→/Home/End) 実装済み。マウス不要で巡回できる。 ✅
5. **HelpTooltip** → focus で出て Escape で消える。Tab 移動中に勝手に開いて閉じるのが少し邪魔だが許容範囲。 ⚠️

### 残課題
- 課題詳細 (`IssueDetail`) に「キーボードでAI生成ボタンを発火させる」ショートカット (`g` キー等) がない。パワーユーザー向けに将来検討余地。
- 検索ボックスへのフォーカスショートカット (`/` キー) がない。一般的なWebアプリで期待されるパターン。

## ペルソナ4: 鈴木 由紀子 (58) / 経理部 (スクリーンリーダー利用)

| 項目 | 内容 |
| --- | --- |
| 職種 | 経理ベテラン (弱視) |
| ITリテラシー | NVDA + Edge の利用に習熟。普段は社内システムで仕訳 |
| 利用シーン | 経理フローの古い記述を見つけて報告 |
| デバイス | Windows ノート + 外付けキーボード、ブラウザ拡大160% |
| ゴール | 自分で報告まで完結できる (誰かに代理入力させない) |
| 困りごと | アイコンだけのボタンが「ボタン」としか読まれない、状態変更が無音 |

### 動線評価
1. **Skip link** → ページ最初の Tab で「メインコンテンツへスキップ」を NVDA が読む。サイドバーをスキップ可能。 ✅
2. **アイコンボタン** → ハンバーガー/閉じる/ヘルプ等すべて `aria-label` を付与。「メニューを開く ボタン」と読まれる。 ✅
3. **ダークモード toggle** → `aria-pressed` を見て「押されています/押されていません」が読まれる。状態変化が伝わる。 ✅
4. **モーダル (WelcomeGuide/ConfirmDialog)** → `role="dialog" aria-modal="true" aria-labelledby` で「ダイアログ ○○」と読まれる。Escapeで閉じる。 ✅
5. **フォームエラー** → `aria-invalid` + `aria-describedby` で「無効な値、タイトルを入力してください」と即時通知。`role="alert"` でフォーカスが当たらなくても読まれる。 ✅
6. **Toast** → エラー時は `role="alert" aria-live="assertive"`、成功時は `role="status" aria-live="polite"`。sr-only で `エラー: / 成功:` プレフィックス。「成功: 課題 ISS-001 を作成しました」と聞こえる。 ✅
7. **拡大160%** → ハンバーガー Drawer はモバイル幅扱いになる可能性あり。要実機検証。 ⚠️

### 残課題
- フロー図 (Mermaid SVG) には `aria-label` / `role="img"` が付いていない可能性 → SVG 描画ライブラリの設定次第。要追加対応。
- 「StatusDot」(点表示) は `role="img" aria-label` を付与済み。テーブル中などで多用された場合、読み上げ過剰にならないか実機確認推奨。
