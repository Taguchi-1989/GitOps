# FlowOps 利用者ペルソナと UX 評価

実在しそうな5つの組織役割でFlowOpsを使ったときに「困らないか/価値が出るか」を評価する。
各ペルソナのリアルな業務シナリオに沿って、改善後の主動線と残課題を整理する。

ブランチ: `claude/improve-ux-accessibility-dTezA`

---

## ペルソナ1: 佐々木 健 (58) / 執行役員 CIO

| 項目 | 内容 |
| --- | --- |
| 役職 | 執行役員 (CIO 兼 DX推進本部長) |
| ITリテラシー | 基本的なPC操作と決裁システム。Git/YAML は知らない |
| 主デバイス | iPhone 15 (移動中)、たまに iPad Pro。PCはほぼ秘書経由 |
| 利用頻度 | 週2-3回、5-10分の隙間時間 |
| ゴール | 全社の業務改善ネタが「いま何件動いていて、何が止まっているか」を即把握。マネージャーに発破をかける |
| 困りごと | 細かい技術用語は読み飛ばす。1画面で要点が見えないと即離脱 |

### 動線評価
1. **iPhoneで朝の通勤中にアクセス** → ハンバーガーDrawer化済みなので画面が圧迫されない。ダッシュボード上部にStatCard 4枚(未対応/作業中/改善案あり/完了)が縦に並ぶ。**3秒で状況把握できる**。 ✅
2. **「未対応の課題」カードをタップ** → 一覧へ。タップ領域44px確保で誤タップしにくい。 ✅
3. **個別課題詳細** → タイトル・ステータス・担当が見える。ただし**経営層が知りたい「停滞日数」「優先度」「部門別集計」は現状の課題詳細にはない**。 ⚠️
4. **専門用語** → 「マージ」「ブランチ」が画面に残る箇所がある。CIO視点では「確定」「下書き」程度の言い換えがあれば理解負荷ゼロにできる。 ⚠️

### 残課題
- **(高)** 経営層向けKPIビュー (停滞7日以上の課題、部門別件数、月次完了数推移)
- **(中)** 「マージ/ブランチ/トリアージ」のさらなる平易化 or 用語ヘルプの常時表示
- **(低)** モバイル向けに「今週確認すべき課題」サマリ通知

## ペルソナ2: 田中 美咲 (28) / 営業企画部 業務改善担当 (現場若手)

| 項目 | 内容 |
| --- | --- |
| 役職 | 営業企画 主任 (非IT職、業務改善ローテで2年目) |
| ITリテラシー | Excel/Notion/Slack。Gitは聞いたことあるレベル |
| 主デバイス | 会社支給ノートPC + iPhone |
| 利用頻度 | 週5回、現場ヒアリング後に都度起票 |
| ゴール | 現場の「これ無駄」を起票して、AIに改善案を出させて上司に承認してもらう |
| 困りごと | 専門用語で固まる、入力ミスで赤くなると焦って手が止まる |

### 動線評価
1. **初回ログイン** → WelcomeGuide 自動表示。「ノード(フロー上のひとつのステップ)」「作業スペース(Gitブランチ)」と言い換え併記済み。**5ステップで概観をつかめる**。 ✅
2. **課題を新規作成** → 「良い課題の書き方」のヒントボックスで現在/期待を書くよう誘導。タイトル空欄送信で赤枠+`タイトルを入力してください`がフィールド直下。Toastも「入力内容を確認してください」と平易。 ✅
3. **作業を開始 → AI改善案を生成** → ConfirmDialog「この操作で何が起きるか」が箇条書きで提示される。何が起きるか不安にならない。 ✅
4. **改善案の差分を確認** → DiffPreviewで`+/-`が色分け+左ボーダー4pxで色覚多様性にも対応済。 ✅
5. **モバイルでの確認** → サイドバーがハンバーガー化、ボタンも44px以上で押しやすい。 ✅

### 残課題
- **(中)** 「作業を開始」した課題の作業スペースを後で見つけ直す導線 (現状はIssue詳細経由のみ。「自分が作業中」フィルタがほしい)
- **(中)** AI生成中の進捗が見えない (待ち時間に不安。プログレスメッセージや所要時間目安が欲しい)

## ペルソナ3: 鈴木 翔 (35) / 情報システム部 中堅SE / 社内コーチ

| 項目 | 内容 |
| --- | --- |
| 役職 | 情シス課長代理。社内DX推進の技術窓口 |
| ITリテラシー | 高い。Git/YAML/TypeScript いける |
| 主デバイス | MacBook Pro + 外部4K。VS Codeと並べる |
| 利用頻度 | 毎日。現場が起票した課題を技術側で整える/フロー定義のYAMLを書く |
| ゴール | 現場が起票した課題を技術的に正しい変更に落とし込む、若手にレクチャーする |
| 困りごと | マウス操作前提のUIはストレス、無駄なクリック多いと萎える、現場に説明する画面の言葉が稚拙だと困る |

### 動線評価
1. **キーボードのみ操作** → Tab で Skip link → メインへ直行可能。サイドバーをスキップできる。 ✅
2. **IssueListタブ切替** → WAI-ARIA tab pattern (←→/Home/End) 実装済み。マウス不要。 ✅
3. **WelcomeGuide** → 矢印キー(←→)でステップ移動、Escapeで閉じる。 ✅
4. **ConfirmDialog** → Escapeでキャンセル、初期フォーカスがキャンセル側で誤Enter回避。 ✅
5. **若手への説明** → 「課題」「フロー」「ステップ」の日本語で統一されており、田中さんへ説明しやすい。 ✅
6. **YAML編集** → 現状UIではYAML直編集はできず、`spec/flows/`配下のファイル編集(VS Code)が前提。情シス的にはこれで十分。 ✅

### 残課題
- **(低)** 検索ボックスへの `/` キーショートカット (一般的なWebアプリで期待される)
- **(低)** AI生成ボタンへの `g` 等キーボードショートカット (パワーユーザー向け)
- **(中)** フロー画面に残る `Issue/Branch/Merge` 英語表記の棚卸し (若手指導時に言葉が揺れる)

## ペルソナ4: 山本 由紀子 (45) / 内部監査部 オペレーション監査担当

| 項目 | 内容 |
| --- | --- |
| 役職 | 内部監査部 課長 (公認内部監査人 CIA保持) |
| ITリテラシー | Excelピボット熟練。SQLは触れる |
| 主デバイス | 会社ノートPC。資料はPDFで保存 |
| 利用頻度 | 四半期1回の定例監査 + ad-hoc監査 |
| ゴール | 「業務フロー変更が職務分掌に従って実施され、適切に記録されているか」を検証 |
| 困りごと | 監査証跡(誰が/いつ/何を/承認者は誰か)が画面で追えないと、APIや生DBを叩く羽目になる |

### 動線評価
1. **過去変更の追跡** → 課題詳細画面で個別の状態遷移は見える(StatusLifecycle)。 ✅
2. **AuditLog API** → `/api/audit?entityType=...&entityId=...` で監査ログ取得可能。 ✅
3. **しかし監査ログ閲覧UIが存在しない** → 現状、監査ログを画面で一覧/絞込/エクスポートする手段がない。**APIを叩いてJSONを目視する必要があり、監査業務として実用上厳しい**。 ❌
4. **承認者・実行者の表示** → AuditLogのactor情報はDBに保存されているが、Issue詳細画面に「誰が承認したか」のサマリが出ない。 ❌
5. **エクスポート** → 監査調書として保存したいが、CSVダウンロード/期間指定検索のUIがない。 ❌

### 残課題
- **(高)** 監査ログ閲覧UI (`/audit` ページ。期間/エンティティ/アクター/アクション絞込、CSVエクスポート)
- **(高)** Issue詳細に「変更履歴タイムライン」(誰が・いつ・何を・traceId付き)
- **(中)** 職務分掌チェック (起票者と承認者が同一でないことの強調表示)

## ペルソナ5: 加藤 真理子 (48) / 情報システム本部 AIガバナンス推進室 (ISO/IEC 42001 内部監査員)

| 項目 | 内容 |
| --- | --- |
| 役職 | AIガバナンス推進室 室長。ISO/IEC 42001 認証取得プロジェクト責任者 |
| ITリテラシー | ITSMS/QMS 監査経験あり。技術詳細より「プロセスが回っているか」を見る |
| 主デバイス | 会社ノートPC + Surface |
| 利用頻度 | 月1回サンプリング監査、年1回フル監査 |
| 認証範囲 | AIが業務プロセス変更を提案する箇所すべて(=FlowOps) |
| ゴール | AIの提案 → 人間の承認 → 適用 → 影響範囲 が追跡可能で、ISO 42001 の「AIシステムのライフサイクル管理」要件を満たすことを検証 |
| 困りごと | AI出力に対する責任の所在・承認エビデンスがバラバラだと不適合判定 |

### ISO/IEC 42001 観点での要件
| 条項 | 要求事項 | FlowOps現状 |
| --- | --- | --- |
| 6.1.2 リスクアセスメント | AIが提案する変更のリスクが事前評価されているか | ConfirmDialog「この操作で何が起きるか」で部分対応。リスク区分の明示はなし |
| 7.4 コミュニケーション | AI判断が誰に通知されたか | Toast/画面表示のみで、通知履歴が残らない |
| 8.2 AIシステム影響評価 | AI改善案の影響範囲が記録されているか | DiffPreviewで差分は見えるが、影響を受ける他フロー/データオブジェクトの自動検出はあるが画面表示は弱い |
| 8.3 データ・モデル管理 | LLMモデルバージョン・プロンプトのトレース | `traceId`でlangfuseに連携あり(`/api/governance/trace/:traceId`) ✅ |
| 9.2 内部監査 | エビデンス抽出 | API はあるが UI がない ⚠️ |
| 10.1 是正処置 | 不適合な提案を却下した記録 | Issue却下機能あり、AuditLogに記録 ✅ |

### 動線評価
1. **AI提案の責任所在追跡** → `traceId` で「どのプロンプト/モデルが、誰のIssueに対して、何を提案し、誰が承認したか」を一気通貫で追える設計は整っている。 ✅
2. **しかし `/api/governance/trace/:traceId` を呼ぶ画面がない** → 監査時にtraceIdを手で取り出し、curl/Postmanで叩く必要がある。**プロセス監査者には実用的でない**。 ❌
3. **AI提案の却下理由** → 却下フィールドはあるが、「なぜ却下したか」をAuditLogに紐づけて検索する画面がない。 ❌
4. **モデルバージョン管理** → ENV経由でモデル指定するため、「いつから・どのモデルで提案していたか」を画面で確認できない。 ⚠️

### 残課題 (ISO 42001 認証取得を見据えた追加要件)
- **(高)** ガバナンス画面 (`/governance/trace/[traceId]`) で AI提案 → 承認 → 適用の連鎖を可視化
- **(高)** AI提案ごとに「リスク区分(低/中/高)」を表示し、高リスクは2段階承認を強制
- **(中)** 利用LLMモデル・プロンプトバージョンの履歴ダッシュボード
- **(中)** 月次AI提案レポート (件数・採択率・却下理由分類)
