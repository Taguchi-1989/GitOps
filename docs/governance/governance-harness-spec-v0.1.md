## 0. 本書の構成（二部構成）

本書は二部からなる。

- **第I部 — コア仕様（オーケストレータ非依存）**: ポリシー・エンジン、決定論ゲート、階層化監査ログ、差し替え可能点。**この部はGitOpsに依存しない**。n8n / Dify / 自前オーケストレータでも同一仕様が成立するよう、外部依存を「アダプタ境界」の外に追い出す。
- **第II部 — GitOps結合仕様（実装先固有）**: 第I部のコアを、既存GitHub Actions GitOpsにどう挟むか。同期挿入（in-line）と前段ゲートウェイ（gateway）の両モードを定義し、既存基盤の制約に応じて選択する。

> **設計原則（最重要）**: コアは「挟まれる側の都合」を一切知らない。GitOpsであろうと別システムであろうと、コアは「入力を受け取り、判定とログを返す」純粋な関数的境界として実装する。これにより、器（オーケストレータ）の乗り換えがコアに波及しない。

# 第I部 — コア仕様（オーケストレータ非依存）

## 1. 目的と非目的

### 1.1 目的

1. 高品質LLM API呼び出しを決定論処理で構造的に削減し、コストを外部ベンダーの価格改定に対して鈍感にする。
2. 機密の外部流出を、ランタイムの確率的判定ではなく**設計時の決定論的分離境界**で防ぐ。
3. すべての判定について**バージョン付き監査証跡**を残し、「いつ・どのポリシーで・誰/何が・なぜ通したか」を事後に再現可能にする。

### 1.2 非目的（やらないことの明示 — スコープ汚染防止）

- **ゼロデイ検出の保証はしない**。出口のコード監査は「既知の危険パターンを確率的に削る」装置であり、未知の悪意あるコードの検出を担保しない（§8.1）。
- **値型機密（値そのものが秘密）の自動安全化はしない**。本ハーネスのマスキングは結合型機密（識別子と属性の結合が秘密）にのみ有効。値型は人間承認ワークフローまでしか支援しない（§8.2）。
- **承認判断そのものの自動化はしない**（初期フェーズ）。判断は人間、記録は機械（§5）。

## 2. アーキテクチャ：独立三層 + 二つの差し替え可能点

```
                 ┌─────────────────────────────────────────┐
   入力 ───────► │  入口ゲート（決定論）                      │
                 │   - 機密混入検査（結合型: Presidio等）     │
                 │   - ポリシー・エンジンを参照              │
                 │   - 判定不能 → 安全側へ倒し例外ハンドラへ  │
                 └──────────────┬──────────────────────────┘
                                │ （機密ゼロ化された入力のみ通過）
                                ▼
                 ┌─────────────────────────────────────────┐
                 │  ★差し替え可能点A: ハーネス（器）         │
                 │   Claude Code / Codex / OpenHands ...     │
                 │   ★差し替え可能点B: モデル呼び出し         │
                 │   LiteLLM 一枚（プロバイダ抽象化）         │
                 └──────────────┬──────────────────────────┘
                                │
                                ▼
                 ┌─────────────────────────────────────────┐
   出力 ◄─────── │  出口ゲート（決定論）                      │
                 │   - コード監査（既知CVE/シークレット/SAST）│
                 │   - 入口判定の独立した検証（監査トリップ） │
                 └──────────────┬──────────────────────────┘
                                │
                                ▼
                 ┌─────────────────────────────────────────┐
                 │  監査ログ（追記専用・階層化・時間減衰）    │
                 │   全ゲート判定を記録。実体は参照、ログは判定│
                 └─────────────────────────────────────────┘
```

### 2.1 層の独立性（不変条件）

- 三層は相互に**プロセス境界 or モジュール境界**で分離する。一層の障害が他層を巻き込まない。
- 差し替え可能点A・Bの交換は、三層のコードに**一切変更を要求しない**。これを破る実装はレビューで却下。

## 3. ポリシー・エンジン

### 3.1 役割

「何が機密か」「判定不能時にどちらへ倒すか」「どのスキルを認可するか」を**単一の真実源**として宣言的に保持する。

### 3.2 P0 要件

| ID | 要件 | 根拠 |
|----|------|------|
| POL-1 | ポリシーは**宣言的記述**（OPA/Rego 推奨）で保持し、コードに埋め込まない | 非エンジニア（合意形成の当事者）がレビュー可能であること、変更がそのまま監査証跡になること |
| POL-2 | ポリシーは**バージョン付き**。各版にハッシュと発効時刻を持つ | 「当時のポリシーで見れば妥当だった」を後から再現するため。今のポリシーで過去を裁かない |
| POL-3 | **判定不能時の既定動作は「安全側へ倒す」**（fail-safe default） | 危険側フォールバック禁止。マスキング済みだから安価側、という分岐を作らない |
| POL-4 | ポリシー変更は**人間ゲート必須**。自動学習ループによる自動書き換えを禁止 | 検証されていない判断が決定論の顔で黙って走る「サイレント仕様変更」を防ぐ |

### 3.3 ポリシーが表現すべき分類軸（最小）

- 機密性タイプ: `combination`（結合型 / マスキング有効） / `value`（値型 / マスキング無効・人間承認必須） / `none`
- リスク等級: `low` / `medium` / `high`（承認ラインと保護厳格度をこれに比例させる）
- 認可スキル集合: タスク種別 → 許可されるスキルのホワイトリスト

## 4. 決定論ゲート（入口 / 出口）

### 4.1 入口ゲート（主たるインターロック）

| ID | 要件 |
|----|------|
| ING-1 | ポリシー・エンジンを参照し、入力中の機密を検出する。結合型はマスキング（Presidio 等の決定論検出）。値型は検出した時点で**外部送出を禁止**し、人間承認フローへ |
| ING-2 | 検出器が判定不能（確信度しきい値未満）の場合、POL-3 に従い**安全側へ倒す**。素通しは禁止 |
| ING-3 | 入口ゲートは LLM を呼ばない。決定論のみ。仕分け対象の生データを外部APIに読ませることは構造を崩すため禁止（仕分け器ローカル必須） |

### 4.2 出口ゲート（独立した監査トリップ）

| ID | 要件 |
|----|------|
| OUTG-1 | ハーネス出力（生成コード等）に対し、既知CVE・混入シークレット・SASTパターンを検査する |
| OUTG-2 | 出口ゲートは入口ゲートと**多様性のある独立検出系**として実装する（同一ロジックの再利用禁止）。入口の誤りを検知する二重化トリップとして機能させる |
| OUTG-3 | 出口検査は「漏洩を止める装置」ではなく「入口の誤り検知＋既知危険の確率的削減」と位置づける。**ゼロデイ担保は謳わない**（§8.1） |

> **役割の明示**: 漏洩防止の主役は入口（§4.1）。出口を緩さの言い訳にする（「出口があるから入口は緩くてよい」）設計は禁止。

## 5. 承認ワークフロー（判断は人・記録は機械）

### 5.1 三点セット

1. **前さばき自動化**: 人に上げる前に、「過去に同型ケースが承認済み かつ ポリシー版が不変」なら前例に従い自動承認し、ログに `auto-approved by precedent #N` と残す。人が判断するのは**前例のない新規パターンのみ**。
2. **リスク等級別の決裁ライン**: 全件を最高権限者に上げない。`low` は現場リーダー即決、`high` のみ上位。等級ごとに「これが揃えばYesと言える」決裁パッケージのテンプレを用意。
3. **判断と記録の分離**: 承認者は「Yes/No + 一言の理由」のみ。監査に耐える完全な証跡（ポリシー版・前例・根拠データ・時刻・主体）への展開は決定論+LLMが担う。

### 5.2 ゴム印化の防止（自動化の裏リスク）

- 自動承認された判断を**事後にランダムサンプリング監査**する層を必ず残す。全件は見ないが、抜き取って「この自動承認は妥当だったか」を人が振り返る。
- これは検査機構自身を PDCA の A（改善）に乗せる仕掛け。検査機構を検査する機構がないと検査機構は腐る。

### 5.3 段階的主権移譲（導入順序）

| Phase | 状態 | 移行条件 |
|-------|------|---------|
| 0 | **全件人手承認**。承認者に張り付いて判断基準を抽出。前例を溜めるだけ溜める | — |
| 1 | 人間判断と機械推奨の**一致率がしきい値超**で、`low` 等級から前例自動承認へ | 一致率 ≥ 設定値（運用で決定） |
| 2 | 自動承認 + §5.2 の事後サンプル監査 | Phase 1 の誤承認率が許容域 |

> 各フェーズ昇格は人間ゲート。自動学習ループにはしない（POL-4 と一貫）。

## 6. 階層化監査ログ

### 6.1 設計原則：実体を入れない

| ID | 要件 |
|----|------|
| LOG-1 | ログには**実体（payload）を入れない**。実体はコンテンツアドレス（中身のハッシュ）で外部ストレージに置き、ログにはハッシュ + 判定のみ記録 |
| LOG-2 | 同一内容は同一ハッシュ → **重複排除が自動で効く**。同じパターンが千回来ても実体は一個 |
| LOG-3 | ログは**追記専用（append-only）**。改竄不能性を担保 |
| LOG-4 | 各エントリに**ポリシー版・判定時刻**を必ず刻む |

### 6.2 重大度による密度の階層化

| 層 | 対象 | 記録粒度 | 保持 |
|----|------|---------|------|
| 薄 | 通過（安全側で素通り） | ハッシュ + ポリシー版 + 判定 = 最小（数百バイト） | 法定保持期間で自動失効。古いものは日次サマリへ畳む |
| 厚 | 例外（人へエスカレーション） | 判断に必要な文脈を全部 | 長期 |
| 完全 | インシデント（ポリシー違反検出） | 再現可能な完全証跡 | 最長期 |

> **思想**: 定常時は間引き、トリップ時だけ高分解能で前後を残す（制御機器のデータロガーと同型）。PDCA が見るのは厚・完全層のみ。薄層は統計集計。

### 6.3 時間減衰

- 薄ログは個別行を捨てて日次サマリ（「この日、通過1.2万件、内訳…」）に畳む。**最近は詳細、過去は粗く**。

## 7. 差し替え可能点

| 点 | 対象 | 抽象化手段 | 不変条件 |
|----|------|-----------|---------|
| A | ハーネス（器） | アダプタ・インターフェース | Claude Code → OpenHands 等の交換が三層に波及しない |
| B | モデル呼び出し | LiteLLM 一枚のプロキシ | ベンダー値上げ時、設定一行で別プロバイダ/ローカルへ振る。直叩き（Claude/Bedrock/Azure 直結）を禁止 |

> **スキルの扱い**: スキルのカタログ（何が使えるか）は外部エコシステムから借りる（器側）。スキルの**認可**（このタスクにこのスキルを使ってよいか）はポリシー・エンジンが決める（自社側）。カタログは外、認可は中。

## 8. 未解決リスク（HAZOP 的先出し — 本ハーネスがカバーしない逸脱）

### 8.1 [P0] 出口コード監査はゼロデイを担保しない

未知の悪意あるコードを LLM で確実に弾けるなら世界が変わる。できない。出口ゲートは既知パターンの確率的削減に格下げして記述すること。**売り文句の柱にしない**。担保できるのは「既知危険の削減」と「判定の監査可能性」であって「安全性そのもの」ではない。

### 8.2 [P0] 値型機密にマスキングは効かない

研究開発の実データ（材料パラメータ等）は値そのものが機密。識別子を剥がしても無害化しない。本ハーネスのマスキング経路を値型に適用してはならない。値型は「ローカルで値→抽象化統計サマリへ変換 → サマリのみ外部へ」の経路に限定し、その変換器はローカル必須。

### 8.3 [P1] 結合の論理は再識別に弱い

「A単独・B単独は無害だから別バッチで出せば安全」は再識別攻撃（外部知識との突合、時間差ログ上での結合）に弱い。組み合わせ爆発で人手承認が追いつかない領域がある。「単独で機密でない」の足し上げが「結合しても機密でない」を保証しない点を、ポリシー設計時に明記。

### 8.4 [P1] 前例自動承認の心理的受容性

技術的には容易だが、現場が「機械が前例で勝手に承認」を呑めず全件人手に戻す組織リスクがある。§5.3 の段階的主権移譲（Phase 0 で溜める→一致率確認→移行）で、現場の信頼を保ったまま負担を下げること。これは技術問題ではなく現場の信頼の問題。

## 9. P0 実装順序（最初に何を作るか）

> **監査ログ配管が最優先**。ポリシーもゲートも初期は中身が空でよい。だが「全判定が記録に残る配管」が最初から通っていないと、後から証跡を遡れず、売り物の核（実証時に担保できる）が永久に手に入らない。空のポリシー・素通しゲートでも、監査配管さえ通っていれば運用しながらルールを足せる。PDCA はこの監査ログの上にしか乗らない。

1. **P0-1**: 監査ログ配管（追記専用・コンテンツアドレス・ポリシー版/時刻刻印）。中身は素通しでよい。
2. **P0-2**: ポリシー・エンジンの骨格（Rego、最小ルール、バージョニング）。
3. **P0-3**: 入口ゲート（結合型 Presidio + fail-safe default）。
4. **P0-4**: 差し替え可能点B（LiteLLM 抽象化）。
5. **P1**: 出口ゲート（既知CVE/シークレット/SAST）。
6. **P1**: 承認ワークフロー Phase 0（全件人手 + 前例蓄積）。
7. **P2**: 前例自動承認 + 事後サンプル監査。

# 第II部 — GitOps 結合仕様（実装先固有）

## 10. 結合モードの選択（決め打ち回避）

第I部のコアは、GitOps への結合方式を二モード持つ。**既存 GitHub Actions が「処理途中に外部判定を同期的に差し込んで結果を待てるか」で選択**する。

| モード | 条件 | 結合方法 |
|--------|------|---------|
| in-line（同期挿入） | ワークフローが同期フック/カスタムステップを許す | ジョブの前後に三層をステップとして挿入。判定結果でジョブを continue / block / escalate |
| gateway（前段配置） | 同期フックが打てない / 既存基盤を触りたくない | 三層を独立サービスとして前段に置き、GitOps への入力を事前にゲートで濾す。GitOps 側は「ゲート通過済み」前提で動く |

> **P0 確認事項**: 現行の GitHub Actions ワークフローが in-line を許すか、田口が確認する。許さなければ gateway へフォールバック。**コアは両モードで不変**（§2.1）なので、この選択は実装後でも切替可能。

## 11. GitOps 固有の挿入点

```
PR / push
   │
   ▼
[入口ゲート] ◄─── ジョブ実行前。機密混入を検査。値型検出でblock＋人間承認issue自動起票
   │
   ▼
[既存ジョブ: Claude Code等によるagentic実行] ◄─ 差し替え可能点A/B
   │
   ▼
[出口ゲート] ◄─── 成果物（差分/生成コード）を検査。CVE/シークレット/SAST
   │
   ▼
[監査ログ] ◄─── 全ステップの判定をcommit SHA・PR番号と紐づけて追記
   │
   ▼
マージ / 承認待ち / ブロック
```

### 11.1 GitOps 固有要件

| ID | 要件 |
|----|------|
| GIT-1 | ゲート判定を **commit SHA / PR 番号** に紐づけて監査ログへ。「どのコミットが、どのポリシー版で、どう判定されたか」を git 履歴と突合可能にする |
| GIT-2 | `high` 等級の検出は、マージをブロックし**承認 issue を自動起票**。決裁パッケージ（§5.1.2）のテンプレを issue 本文に展開 |
| GIT-3 | 前例自動承認（Phase 1+）は、過去の承認済み PR を前例ソースとして参照。`auto-approved by precedent PR #N` をコミットステータスに記録 |
| GIT-4 | 三層は GitOps 固有 API（GitHub Actions context 等）に直接依存しない。依存はアダプタ層に隔離し、第I部コアへ漏らさない（移植性の担保） |

---

## 12. 受け入れ基準（Definition of Done）

- [ ] 監査ログ配管が通り、素通し状態でも全判定が追記され、ポリシー版・時刻・ハッシュが刻まれている（P0-1）
- [ ] ポリシー変更が人間ゲートを通らずに反映される経路が存在しない（POL-4）
- [ ] 判定不能時に安全側へ倒れることを示すテストが通る（POL-3 / ING-2）
- [ ] 差し替え可能点B で、設定一行のプロバイダ切替が三層コード無変更で成立する（LiteLLM）
- [ ] in-line / gateway いずれのモードでもコア三層が無変更で動作する（§2.1 / §10）
- [ ] §8 の全リスクが、関係者にレビューされ受容判断が記録されている
