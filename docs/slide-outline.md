# FlowOps 紹介スライド 骨子

> 対象: プロジェクト関係者 / 社内PRプレゼン用
> 形式: HTMLスライド（シングルファイル、ブラウザで閲覧可能）

---

## スライド構成（全12枚想定）

### Slide 1: タイトル
- **FlowOps - GitOps for Business**
- 業務フローを「コード」として管理し、非エンジニアでも改善できるプラットフォーム
- サブタイトル: 業務プロセスの透明性と継続的改善を実現

---

### Slide 2: 課題提起 (Why - なぜ必要か)
- **現状の問題点**
  - 業務フローがExcel/PowerPointに散在し、正本が不明
  - 「最新版」が複数存在し、誰がいつ変更したか追えない
  - 改善提案のフィードバックループが遅い（口頭・メール依存）
  - エンジニアしか触れないシステム vs 業務を知る現場の分断
- **結果**: 業務プロセスの改善が属人的・場当たり的になる

---

### Slide 3: FlowOpsとは (What - 何をするツールか)
- **一言で**: 業務フローのGitHub
- 業務フロー定義（YAML）をSingle Source of Truthとして管理
- Issue起票 → LLM自動提案 → レビュー → マージ のサイクル
- 非エンジニアでもWebUIから改善提案ができる
- 全ての変更に履歴・監査ログが残る

---

### Slide 4: 誰が使うのか (Who - ステークホルダー)

| ロール | FlowOpsでの役割 |
|--------|-----------------|
| **業務担当者** | Issue起票（問題報告・改善要望）、提案レビュー |
| **プロセスオーナー** | フロー承認、マージ判断、優先度決定 |
| **システム管理者** | 環境構築、LLMプロバイダー設定 |
| **LLM (AI)** | Issue内容からYAML修正パッチを自動生成 |

- ポイント: **コードを書けなくてもフロー改善に参加できる**

---

### Slide 5: いつ使うのか (When - 利用タイミング)
- **日常**: 業務中に気づいた問題・改善点をIssueとして報告
- **定期**: 月次/四半期のプロセスレビュー会議の素材として
- **イベント駆動**: システム変更時のフロー影響分析
- **監査時**: 変更履歴・監査ログの参照

---

### Slide 6: どこで動くのか (Where - 環境・構成)
- **技術スタック概要**
  - Next.js 14 (フルスタック) + SQLite + ローカルGitリポジトリ
  - LLM: OpenAI / Anthropic / Gemini / Groq / Ollama (ローカル対応)
- **デプロイ形態**
  - 開発: ローカル (`npm run dev`)
  - 本番: Vercel / Docker / オンプレ (Node.js 18+)
- **データの所在**
  - YAML正本: `spec/flows/*.yaml` (Gitリポジトリ内)
  - Issue/提案: SQLite (Prisma ORM)
  - 監査ログ: SQLite (全操作を記録)

---

### Slide 7: コアアーキテクチャ図
- README記載のアーキテクチャ図をビジュアル化
  ```
  UI Layer → REST API → Core Logic → Persistence
  (Next.js)             (Parser/Git/LLM/Patch/Issue/Audit)
                                      (SQLite + Git)
  ```
- 設計原則:
  - **Git-first**: DB更新はGit commit成功後のみ
  - **Zod境界検証**: 全入出力をスキーマで検証
  - **Mutex Lock**: Git操作の排他制御

---

### Slide 8: ワークフロー (How - どう使うか①)
- **5ステップの改善サイクル**:
  1. **Issue作成**: 問題・改善点を報告（WebUIから）
  2. **作業開始**: ブランチ自動作成 (`cr/ISS-001-fix-typo`)
  3. **LLM提案生成**: AIがYAML修正案（JSON Patch）を自動生成
  4. **パッチ適用**: 差分プレビュー確認 → 適用 → Git commit
  5. **マージ＆クローズ**: mainへマージ → Issue完了
- 図: フローチャートで視覚的に表現

---

### Slide 9: デモ画面イメージ (How - どう使うか②)
- **ダッシュボード**: Issue統計、最近のIssue、フロー一覧
- **フロービューア**: Mermaid.jsによるフローチャート可視化
- **Issue詳細**: ステータス遷移、LLM提案カード、差分プレビュー
- スクリーンショットまたはモックアップを配置

---

### Slide 10: 安全性と信頼性
- **セキュリティ**
  - パストラバーサル対策 (flowId検証)
  - LLM入出力の境界制御 (秘密情報を渡さない / Zod検証)
  - 禁止パス検出 (`/id` 変更不可)
- **信頼性**
  - Git排他制御 (Mutex Lock, タイムアウト30s, 自動解除60s)
  - baseHashによる陳腐化検知（楽観的ロック）
  - 監査ログによる完全な操作追跡
- **テスト**: Vitest による自動テスト (Parser / Patch / Hash / Issue / Git Lock)

---

### Slide 11: コスト感 (How Much - 導入・運用コスト)
- **インフラコスト**
  - SQLite + ローカルGit → 追加DBサーバー不要
  - Ollama (ローカルLLM) 選択時はAPI費用ゼロ
  - Vercelフリープランでもデプロイ可能
- **LLM API費用** (参考)
  - OpenAI gpt-4o: 提案1件あたり数円程度
  - ローカルLLM (Ollama): 無料
- **導入工数**
  - セットアップ: `npm install` → `.env` 設定 → `prisma db push` → 起動
  - 所要時間: 環境構築は約10分
- **学習コスト**
  - エンドユーザー: WebUI操作のみ（Git知識不要）
  - 管理者: Node.js + Git の基本知識

---
