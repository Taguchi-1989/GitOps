SLIDE 2

起動手順（最短）

この4コマンドでローカルデモが開始できます

npm install cp .env.example .env.local npx prisma db push npm run dev

アクセス

  • http://localhost:3000
  • APIヘルス確認: /api/health

.env.local 最低限

  • DATABASE_URL=file:./dev.db
  • LLM_PROVIDER=openai
  • LLM_API_KEY=...
準備が終わったらすぐデモに入れる
SLIDE 3

デモシナリオ（5-10分）

前半

  • /api/flows でフロー一覧確認
  • /issues/new でIssue作成
  • /start で作業ブランチ作成

後半

  • /proposals/generate で提案生成
  • /proposals/:id/apply でYAML更新+commit
  • /merge-close でmainマージ+Issue完了

見せる順番を固定すると、聞き手が「何が自動化されているか」を追いやすくなります。

Issue中心のGitOpsフローを可視化
SLIDE 4

どこにデータを上げるべきか

正本（最重要）

  • spec/flows/*.yaml がSSOT
  • 最終的にGit commit / pushで共有
  • YAMLが業務フローの唯一の真実

補助データ

  • 辞書: spec/dictionary/*.yaml
  • 運用DB: prisma/dev.db（Issue/Proposal/Audit）
  • 証跡: 外部ストレージに保存しURL管理

注意: ローカルSQLiteはチーム共有されません。共有が必要ならDB移行方針を別途定義。

正本はYAML、DBは運用メタ情報
SLIDE 5

拡張時の運用ルール

開発フロー

  • 変更はまずIssue化する
  • /start でブランチを作る
  • 提案をレビューしてから適用する
  • 適用後にテスト・型チェックを実行

検証コマンド

npm run test npm run typecheck npm run lint
レビューと検証を入れると品質が安定
SLIDE 6

説明トーク例（そのまま読める版）

  • 「Flowの正本はYAMLです。UIはYAMLを安全に更新する入口です。」
  • 「Issueを起点にブランチ作成、LLM提案、Patch適用、マージまでを一本化しています。」
  • 「適用時にYAML更新とGit commitを自動実行し、履歴を残します。」
  • 「保存先は、正本=spec/flows、運用データ=SQLite、証跡=外部URL管理です。」
デモ中に迷わない説明スクリプト
SLIDE 7

実装との対応（根拠ファイル）

主な処理

  • フロー読込/保存: src/lib/flow-service.ts
  • Issue作成: src/app/api/issues/route.ts
  • 提案生成: src/app/api/issues/[id]/proposals/generate/route.ts
  • 提案適用: src/app/api/proposals/[id]/apply/route.ts

Git / DB

  • Git操作: src/core/git/manager.ts
  • マージクローズ: src/app/api/issues/[id]/merge-close/route.ts
  • DBスキーマ: prisma/schema.prisma
説明の信頼性を上げる参照先
