GitOps for Business

FlowOps

業務フローを「コード」として管理し、
非エンジニアでも改善できるプラットフォーム

業務プロセスの透明性と継続的改善を実現
Why

現場が抱える4つの課題

1

正本が不明

業務フローがExcel/PPTに散在し、どれが最新か分からない

2

変更が追えない

「最新版」が複数存在し、誰がいつ何を変更したか不透明

3

改善ループが遅い

フィードバックが口頭・メール依存で、提案から反映まで数週間

4

現場とシステムの分断

エンジニアしか触れないシステム vs 業務を知る現場の壁

⚠️

結果

業務プロセスの改善が
属人的・場当たり的になる

What

FlowOps = 業務フローの GitHub

  • 業務フロー定義（YAML）を
    Single Source of Truthとして管理
  • Issue起票 → LLM自動提案 → レビュー → マージの改善サイクル
  • 非エンジニアでもWebUIから改善提案

YAML = 正本 (SSOT)

DBはあくまで派生データ。spec/flows/*.yaml が唯一の正解

spec/flows/order-process.yaml
id: order-process
title: 受注処理フロー
layer: L1

nodes:
  receive_order:
    type: process
    label: 受注受付
    role: 営業
    system: CRM
    # Git管理された定義

When

利用タイミング

📅 日常

業務中に気づいた問題・改善点をIssueとして即座に報告。記憶が新鮮なうちにアクション。

📈 定期レビュー

月次/四半期のプロセスレビュー会議で、蓄積されたIssueや変更履歴を素材として活用。

⚡ イベント駆動

システム変更や組織改編時。変更によるフローへの影響範囲を分析。

🔍 監査対応

「誰が/いつ/何を/なぜ」変更したか、Git履歴と監査ログで完全なトレーサビリティを確保。

設計原則

Git-first

DB更新はGit commit成功後のみ。不整合を原理的に排除。

Zod Boundary

全ての入出力をスキーマ検証。LLMのハルシネーションも水際で防ぐ。

Optimistic Lock

パッチ生成時のYAMLハッシュ(baseHash)で陳腐化を検知。

Workflow

5ステップの改善サイクル

1

Issue 作成

WebUIから問題報告。Evidence添付可能。

2

作業開始

ブランチ自動作成 (e.g. cr/ISS-001)

3

LLM 提案生成

AIがYAML修正案(JSON Patch)を生成。

4

パッチ適用

差分プレビュー確認 → 適用 → Commit。

5

マージ & Close

mainへマージ。全履歴がGitに残る。

Core API Endpoints

POST /api/issues Create
POST /api/issues/:id/start Branch
POST .../proposals/generate LLM

Security

堅牢なセキュリティ設計

Security Measures

🔒

Path Traversal

IDは英数字限定。不正パスアクセスを遮断。

🤖

LLM Input Scope

spec/flowsのみ渡す。.envや機密情報は渡さない。

✅

Output Validation

Zodスキーマ + 辞書チェックで出力を厳格検証。

📝

Audit Log

全操作をログ化。いつ誰が変更したか完全追跡。

Reliability

Git Mutex Lock

Git操作中は排他ロック。タイムアウト30s/自動解除60s。

baseHash Check

編集競合をハッシュ値で検知し、古い状態へのパッチ適用を防止。

Automated Test

VitestでParser/Patch/Lock機構を保護。
