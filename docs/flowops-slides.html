GitOps for Business

FlowOps

業務フローを「コード」として管理し、
非エンジニアでも改善できるプラットフォーム

業務プロセスの透明性と継続的改善を実現

Why — なぜ必要か

現場が抱える4つの課題

1

正本が不明
業務フローがExcel/PowerPointに散在し、どれが最新か分からない

2

変更が追えない
「最新版」が複数存在し、誰がいつ何を変更したか不透明

3

改善ループが遅い
フィードバックが口頭・メール依存で、提案から反映まで数週間

4

現場とシステムの分断
エンジニアしか触れないシステム vs 業務を知る現場の壁

⚠️

結果

業務プロセスの改善が
属人的・場当たり的になる

What — 何をするツールか

FlowOps = 業務フローの GitHub

  • 業務フロー定義（YAML）をSingle Source of Truthとして管理
  • Issue起票 → LLM自動提案 → レビュー → マージの改善サイクル
  • 非エンジニアでもWebUIから改善提案ができる
  • 全ての変更に履歴・監査ログが残る

YAML = 正本 (SSOT)

DBはあくまで派生データ。spec/flows/*.yaml が唯一の正解

spec/flows/order-process.yaml

id: order-process
title: 受注処理フロー
layer: L1

nodes:
  receive_order:
    type: process
    label: 受注受付
    role: 営業
    system: CRM

設計原則

Git-first

DB更新はGit commit成功後のみ。不整合を原理的に排除

Zod 境界検証

全ての入出力をスキーマで検証。LLM出力も例外なし

Mutex Lock

Git操作の排他制御。タイムアウト30s、自動解除60s

baseHash 楽観ロック

パッチ生成時のYAMLハッシュで陳腐化を検知

How — どう使うか

5ステップの改善サイクル

1

Issue 作成

WebUIから問題・改善点を報告。Evidence（証跡）も添付可能

2

作業開始

ブランチ自動作成 cr/ISS-001-fix-typo

3

LLM 提案生成

AIがYAML修正案（JSON Patch）を自動生成。差分プレビュー付き

4

パッチ適用

差分プレビューを確認 → 適用 → Git commit

5

マージ & クローズ

main へマージ → Issue 完了。全履歴がGit + 監査ログに残る

API エンドポイント

POST/api/issues
POST/api/issues/:id/start
POST/api/issues/:id/proposals/generate
POST/api/proposals/:id/apply
POST/api/issues/:id/merge-close

鉄則: DB状態の更新は Git commit が成功した後にのみ実行される

重複統合: 同じ指摘が複数来てもmerge-duplicateでシステムレベルで統合
