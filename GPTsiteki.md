# 企業内AI活用基盤 要件定義書（たたき台）

## 1. 文書の目的

本書は、企業内で散在する文書・図面・表計算・検討資料・報告資料を、AIエージェントが安全かつ実用的に活用できる状態へ段階的に移行するための要件定義のたたき台である。

本取り組みは、既存の Word / Excel / PowerPoint をそのまま AI 化することを目的としない。目的は、業務上の意思決定に必要な情報を、意味・出所・更新時点・権限・妥当性の観点で再構造化し、AI が参照・検証・提案・一部実行できる基盤を整備することである。

---

## 2. 背景認識

### 2.1 現状課題

* 情報が Word / Excel / PowerPoint / PDF / 画像 / メール / 会議資料などに散在している
* 同一概念が部署ごとに異なる名称で管理されている
* 誰が作成した情報か、いつ時点の情報か、何を根拠にした数値かが不明瞭な資料が多い
* OCR 以前に、図・表・レイアウト・注記・単位・版番号・承認履歴を含む意味理解が不足している
* RAG 的な検索だけでは、正しさ・責任所在・最新版判定・アクセス権制御が不十分
* Office 文書作成の自動化だけでは、業務目的や判断基準そのものは整理されない

### 2.2 目指す方向

* 文書中心ではなく「業務目的」「意思決定」「根拠データ」「承認条件」中心に情報構造を再設計する
* ファイルを読む AI ではなく、意味づけされたデータ・根拠・ルール・履歴を扱う AI に移行する
* AI の出力は提案で終わらせず、根拠・参照元・信頼度・検証結果を必ず伴わせる
* 更新・申請・発注・通知などの実行は、明確な権限と承認境界の中でのみ許可する

---

## 3. あるべき姿（To-Be）

### 3.1 業務のあるべき姿

利用者は「資料を作る」ではなく「何を判断したいか」「何を確認したいか」を自然言語で入力する。
システムは以下を実施する。

* 必要な情報源候補を選定する
* OCR / 構造抽出 / 意味抽出済みデータを参照する
* 最新性・作成者・承認状態・版数・権限を確認する
* 計算根拠や整合性を自動検証する
* 根拠付きで回答・要約・ドラフト・差分比較・論点整理を返す
* 実行系処理は人の承認後に限定実行する

### 3.2 データのあるべき姿

各データ・文書・画像・表・図面・帳票・議事録に対し、最低限以下のメタデータを保持する。

* 一意ID
* タイトル
* 文書種別
* 所有部門 / 所有者
* 作成者 / 承認者
* 作成日時 / 更新日時 / 有効期限
* 版番号 / 改訂履歴
* 出所システム
* 権限制御属性
* 業務ドメイン
* 関連案件 / 設備 / 製品 / 顧客 / 工程
* 根拠元（引用・参照リンク）
* 抽出信頼度
* 検証状態（未検証 / 機械検証済 / 人手確認済 / 正式版）

### 3.3 AI のあるべき姿

* 参照した根拠を明示できる
* 古い情報や権限外情報を除外できる
* 計算値・単位・表間整合・版差異を検証できる
* 回答不能時には不足情報・確認依頼事項を返せる
* 実行時には操作ログと監査証跡を残せる

---

## 4. 基本方針

1. 全社一括ではなく、重要業務から段階導入する
2. OCR 導入だけで終わらせず、構造化・意味付け・検証まで含める
3. 生成AIの前に、真実のソースと権限モデルを整える
4. 文書を直接真実と見なさず、意味層・根拠層・履歴層を分離して持つ
5. AI の自由実行を避け、提案→検証→承認→実行の境界を設ける
6. 評価（精度・根拠率・誤答率・コスト・レイテンシ・監査性）を先に定義する

## 5. スコープ

### 5.1 対象業務（初期候補）

* 技術検討資料の横断検索と要約
* 過去検討書・報告書・標準書からの根拠抽出
* 図表を含む PDF / 画像の OCR・構造抽出
* 計算書・表計算の整合性チェック
* 版差分比較
* ドラフト資料の自動生成

### 5.2 初期対象外

* ERP / 基幹システムへの自動更新
* 顧客向け自動送信
* 人事・評価・法務の高機密判断
* 完全自律型の無承認実行

## 7. 機能要件

### 7.1 文書取得

* PDF / Word / Excel / PowerPoint / 画像を取り込めること
* 原本を保持したまま処理できること
* 版差分を識別できること

### 7.2 OCR・構造抽出

* 文字、表、見出し、箇条書き、脚注、図、式を抽出できること
* 座標情報とページ情報を保持できること
* 低品質画像に対しても再処理手段を持つこと

### 7.3 意味抽出

* 文書種別判定ができること
* 固有表現を抽出できること
* 同義語・略語を正規化できること
* 業務ドメイン分類ができること

### 7.4 妥当性検証

* 抽出値に信頼度を持てること
* ルールベース検証ができること
* 人手確認対象を自動マーキングできること
* 計算値再計算または再検証ができること

### 7.5 検索・回答

* メタデータ条件付き検索ができること
* 権限に応じた検索結果制御ができること
* 回答に根拠を添付できること
* 最新版優先・正式版優先ができること

### 7.6 監査・運用

* 参照ログ、回答ログ、操作ログを保存できること
* 評価データセットを蓄積できること
* 誤答や不適切抽出をフィードバックできること

## 8. 情報分類・セキュリティ・ガバナンス強化要件

### 8.1 基本思想

本基盤では、すべての情報を同一のデータとして扱わない。特に、配合条件、レシピ、危険パラメータ、製造条件、設計ノウハウ、しきい値、歩留まり条件、代替困難な経験則などは、通常文書と分離して扱う必要がある。

したがって、データは少なくとも以下の3層に分ける。

* 秘匿原本層：実数値・実レシピ・危険条件・未公開条件を含む原本
* 抽象化意味層：レンジ化、匿名化、カテゴリ化、関係性化された意味表現
* 成果物層：Word / Excel / PowerPoint / PDF / レポート等の人間向け出力

AI が通常参照するのは原則として抽象化意味層とし、秘匿原本層へのアクセスは限定された計算環境・限定権限・限定目的にのみ許可する。

### 8.2 情報分類レベル

最低限、以下の機密区分を定義する。

* L0: 公開情報
* L1: 社内一般
* L2: 部門限定
* L3: 機密情報
* L4: 高機密情報
* L5: 極秘情報 / 危険パラメータ / 中核ノウハウ

各レベルに対し、以下を定義すること。

* 閲覧可能者
* 保存可能領域
* 持ち出し可否
* 生成AI利用可否
* 外部送信可否
* OCR / 抽出可否
* 抽象化必須有無
* 人手承認要否
* ログ保存要件

### 8.3 抽象化ポリシー

L4〜L5 情報は、そのまま検索・生成・要約に使わない。必要に応じて以下の抽象化を行う。

* 数値の秘匿（マスキング、レンジ化、ビン化、相対化）
* 固有名称の秘匿（設備名、配合名、顧客名、材料名のトークン化）
* 条件の抽象化（具体レシピではなく制約条件、必要特性、依存関係として表現）
* 結果のみ利用（入力値は秘匿し、成立条件・失敗条件・傾向のみ残す）
* 差分のみ利用（基準値を消し、増減や閾値関係のみ残す）

抽象化前原本と抽象化後データは、必ず追跡可能に紐付くこと。ただし通常利用者は逆引きできない構造とする。

### 8.4 実行環境分離

情報レベルに応じ、処理環境を分離する。

* 一般領域：L0〜L2 を扱う通常業務領域
* 制限領域：L3〜L4 を扱う認証強化領域
* 隔離領域：L5 を扱う閉域 / ローカル限定 / 物理制約付き領域

隔離領域では、以下を検討対象とする。

* インターネット非接続
* 外部API利用禁止
* モデル持込または閉域推論
* USB 等の持出媒体制御
* 画面転送・コピー制御
* 管理者を含むアクセス制限
* 実行ログの厳格保存
* 承認済み成果物のみ外部搬出

### 8.5 アクセス制御方針

アクセス制御は、単なるフォルダ権限ではなく属性ベースで行う。
最低限以下の属性を持つ。

* 機密区分
* 所有部門
* 所有者
* 利用目的
* 閲覧資格
* 承認状態
* 有効期限
* 輸出 / 持出制約
* AI利用可否
* 抽象化必須有無

### 8.6 監査・不正検知

以下を必須とする。

* 誰が見たか
* いつ見たか
* 何を検索したか
* どの根拠にアクセスしたか
* 何を出力したか
* どのラベルで持ち出したか
* 大量閲覧 / 大量ダウンロード / 異常時間帯アクセスの検知
* 権限変更履歴の保存

## 9. 非機能要件

### 8.1 セキュリティ

* 文書単位・属性単位でアクセス制御できること
* 機密区分に応じて利用制限を設けられること
* 通信・保存データを暗号化できること

### 8.2 可観測性

* どの文書を参照したか追跡できること
* どのルールで除外・採用したか記録できること
* 評価指標を継続監視できること

### 8.3 拡張性

* OCR エンジンや LLM を差し替え可能であること
* UI だけでなく API / Agent から利用可能であること
* 将来的な MCP / ツール連携を阻害しないこと

### 8.4 運用性

* 文書追加時に再処理できること
* モデル更新時に差分検証できること
* 失敗時に再実行可能であること

## 10. データモデル要件（強化版）

### 10.1 設計原則

データ表現は「単なる1つの巨大JSON」ではなく、以下の考え方で分離する。

* ヘッダ：管理・分類・権限・版・出所などの共通属性
* 本文：OCRテキスト、表、図、式、説明文などの内容本体
* 意味オブジェクト：エンティティ、関係、条件、根拠、判断、制約
* 検証オブジェクト：妥当性チェック結果、再計算結果、例外理由
* 由来オブジェクト：どこから来たか、誰が作ったか、どう変換されたか

### 10.2 データの単位

最小単位は「ファイル」ではなく、以下を併用する。

* Document：文書全体
* Fragment：ページ、段落、表、図、セル範囲、数式、注記
* Semantic Object：設備、配合、条件、材料、手順、結論、制約
* Evidence Link：主張と根拠の紐付け
* Transformation Event：OCR、抽出、変換、要約、検証、承認などの処理イベント

### 10.3 推奨データ構造

保管構造は次のハイブリッド構成を推奨する。

* オブジェクトストレージ：原本ファイル保存
* JSON/JSON-LD：メタデータ・意味オブジェクト・由来情報
* 検索インデックス：全文検索・セマンティック検索
* グラフ表現：関係性・由来・依存関係・根拠連鎖
* 監査ログストア：参照・出力・承認・権限変更履歴

### 10.4 ハッシュの位置づけ

ハッシュは重要だが、ハッシュだけでは意味や権限は表せない。
したがってハッシュは以下の用途に限定して用いる。

* 原本同一性確認
* 改ざん検知
* 抽象化前後の対応関係管理
* 版差分検出補助
* 署名・承認証跡との連携

意味・分類・権限・根拠は、別のメタデータとして保持する。

### 10.5 典型レコード構造（概念）

1つのデータオブジェクトは概念的に以下を持つ。

* object_id
* parent_id
* object_type
* sensitivity_level
* source_hash
* owner
* created_at / updated_at
* version
* access_policy_ref
* content_ref
* semantic_tags
* provenance_ref
* validation_ref
* export_policy

### 10.6 親子関係と関係線

ユーザーのイメージ通り、親子に連なっていくツリーは有効だが、それだけでは不足する。
理由は、同じ表が複数文書から参照されたり、1つの結論が複数根拠を持ったり、1つの抽象化データが複数原本由来だったりするためである。
したがって、基本は親子構造を持ちつつ、横断関係はグラフで管理する。

### 10.7 抽象化データの持ち方

抽象化データは原本の単純コピーではなく、別オブジェクトとして保持する。
その際、以下を持つ。

* 抽象化ポリシー種別
* 元データ参照ID
* 逆引き可否フラグ
* 再識別リスク評価
* 持出可否
* 利用可能目的

### 10.8 出力との接続

PowerPoint / Word / Excel はデータ本体ではなく、レンダリング先・成果物として扱う。
したがって、出力ファイルには以下を残す。

* 参照元オブジェクトID
* 生成日時
* 生成者 / 生成システム
* 出力用ラベル
* 引用元一覧
* 再生成可否

## 14. 最初のPoCテーマ案

強化しました。
今回の論点で一番大事なのは、**「全部を同じデータとして扱わない」**ことです。

あなたが言っている
「本当に危ない配合条件やレシピは、そのままAIに食わせない」
「抽象化した意味データだけを通常利用に回す」
「原本は閉域・ローカル・厳格監査で扱う」
は、かなり正しい設計思想です。NIST の Zero Trust は境界防御よりもユーザー・資産・リソース単位で守る考え方を取っており、Purview の sensitivity labels もデータの分類と保護を前提にしています。 ([NISTコンピュータセキュリティリソースセンター][1])

今回、要件書のたたき台には次を追加しました。
機密区分、抽象化ポリシー、隔離領域、属性ベースのアクセス制御、監査、不正検知、そして「データはヘッダ・本文・意味オブジェクト・検証オブジェクト・由来オブジェクトに分ける」という強化版データモデルです。

設計の考え方としては、3層に分けるのがよいです。

**1. 秘匿原本層**
実数値、レシピ、危険パラメータ、製造条件そのものを持つ層です。ここは通常の検索・生成の対象にせず、極小人数、厳格ログ、場合によっては閉域・持ち出し制限付きで扱います。高機密ワークロード向けには、Azure Confidential Computing や Google Confidential Space、AWS Nitro Enclaves のように、処理中データを隔離する仕組みがあります。特に Nitro Enclaves は外部ネットワークなし・永続ストレージなしの制約付き実行環境として提供されています。もっと強い要件なら、これでも足りず物理分離や完全閉域が必要です。 ([Microsoft Learn][2])

**2. 抽象化意味層**
通常のAIが見るべきなのはここです。
数値はそのままではなく、レンジ、傾向、閾値関係、依存関係、成立条件、失敗条件、相対比較に変換します。
たとえば「A=23.7, B=0.91」ではなく、
「Aは高域」「Bは下限付近」「この組み合わせでは安定化条件Xが必要」
のように持つイメージです。
Google の Sensitive Data Protection も、検出・分類・de-identification を前提にしています。 ([Google Cloud Documentation][3])

**3. 成果物層**
Word、Excel、PowerPoint、PDF はここです。
つまり、これらは“真実のソース”ではなく、**レンダリング先・配布先・説明媒体**です。
出力ファイル側には、どのデータを使って生成したか、誰が出したか、どのラベルで持ち出せるか、を残すべきです。Purview のラベルや MIP SDK のメタデータは、この考え方と相性がいいです。MIP SDK ではラベル情報をキー・バリューのメタデータとしてファイルやレコードに保持できます。 ([Microsoft Learn][4])

あなたの「ハッシュ化したのを使うのか」「JSONヘッダとして持つのか」という問いに対しては、答えはこうです。

**ハッシュは使う。ただし、ハッシュだけでは足りない。**
ハッシュでできるのは、原本同一性確認、改ざん検知、版差分追跡、抽象化前後の対応管理です。
でも、機密区分、利用目的、アクセス権、意味、根拠、由来までは表せません。
なので、**ハッシュは“証跡”用、意味や権限は別メタデータ**で持つのが正解です。これは Purview や MIP のラベル設計、W3C PROV の provenance モデルとも整合します。 ([Microsoft Learn][4])

データの持ち方は、**巨大な1個のJSON** にしない方がいいです。
おすすめはハイブリッドです。

* 原本ファイルはオブジェクトストレージ
* メタデータと意味オブジェクトは JSON / JSON-LD
* 全文検索とセマンティック検索は検索インデックス
* 関係性と由来はグラフ
* 操作履歴は監査ログストア

JSON-LD は既存のJSONから linked data に滑らかに拡張でき、W3C PROV は「誰が・何を・どう作ったか」という provenance 表現の標準です。OpenLineage も lineage metadata の相互運用仕様として使えます。 ([W3C][5])

なので、あなたのイメージは
**「親子ツリー + 横断リンクのグラフ」**
にすると近いです。
親子だけだと、1つの結論に複数根拠がぶら下がる、複数文書が同じ表を参照する、抽象化データが複数原本から来る、という実務の関係を表しきれません。
だから、基本は親子構造、由来や参照はグラフ、がよいです。Purview や OpenLineage も lineage を関係として扱っています。 ([Microsoft Learn][6])

イメージとしては、1オブジェクトはこのくらいです。

```json
{
  "object_id": "obj-2026-000123",
  "parent_id": "doc-2026-000045",
  "object_type": "semantic_constraint",
  "sensitivity_level": "L5",
  "source_hash": "sha256:...",
  "owner_org": "ProcessTech",
  "created_at": "2026-03-08T10:00:00+09:00",
  "version": "3.1",
  "access_policy_ref": "policy-L5-local-only",
  "content_ref": "store://semantic/obj-2026-000123.json",
  "semantic_tags": ["recipe", "constraint", "safety-critical"],
  "provenance_ref": "prov://run-88421",
  "validation_ref": "val://check-1092",
  "export_policy": "abstracted-only"
}
```

この `content_ref` の先に、
OCR本文、表、式、抽象化後の条件、検証結果、根拠リンク、承認状態を別オブジェクトで持たせる感じです。
つまり「全部入りJSON」ではなく、**ヘッダで管理し、中身は分割して参照する**設計です。JSON-LD を使うと、この参照関係を自然に表現しやすいです。 ([W3C][5])

セキュリティ・ガバナンスのルールとしては、最低でも次の軸が必要です。

* 機密区分
* 原本利用可否
* AI利用可否
* 抽象化必須有無
* 閉域限定かどうか
* 持ち出し可否
* 逆引き可否
* 利用目的
* 承認状態
* 監査レベル

このとき、アクセス制御はフォルダ権限だけでは足りず、**属性ベース**で設計した方がいいです。Zero Trust の考え方とも一致します。 ([NISTコンピュータセキュリティリソースセンター][1])

実務上の結論を一言で言うと、こうです。

**原本は守る。AIには抽象化した意味を見せる。成果物はそこから生成する。**

そしてデータ構造は、
**原本保管 + メタデータヘッダ + 意味オブジェクト + 検証オブジェクト + provenance/lineage + 監査ログ**
の分離構造にするのが、一番ぶれにくいです。
Office は最後の出口であって、中核のデータ構造ではありません。 ([W3C][7])

必要なら次に、この強化版をそのまま
**「データオブジェクト設計書 v0.1」**
として、JSONスキーマ風に落とします。
